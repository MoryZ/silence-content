# Proposal: tournament_score_record CRUD

## Intent

为赛事得分明细记录（tournament_score_record）建立 CRUD 能力，支持按场次/片/周期维度记录参赛单位的得分来源，为排行榜计算与得分回溯提供数据依据。

影响模块：silence-content-service-api、silence-content-service

## Scope

### 接口安全

- 鉴权级别：登录用户（系统内部调用，由网关/框架统一拦截）

### 性能承诺

- 创建接口：P95 < 100ms
- 查询接口（按参赛单位+玩法）：P95 < 100ms（结果分页，禁止全量扫描）

### 下游依赖

无 FeignClient 外部服务依赖，仅涉及本服务 DB 写入。

### 异常场景

1. `eventGameId` 不存在 → IllegalArgumentException (400)
2. `scoreType` 传入非法枚举值 → IllegalArgumentException (400)
3. 目标得分记录 id 不存在 → ResourceNotFoundException (404)

## Non-Goals

- 不在本提案中定义得分聚合/汇总逻辑（汇总由 RegistrationEndJob / DailyCacheRebuildJob 负责）。
- 不引入外键约束或联合唯一索引。
- 不处理得分撤销或历史修正逻辑。
- 不涉及排行榜重算触发。

## Approach

### 分层策略

按项目标准 CRUD 分层：
1. `silence-content-service-api`：定义 `TournamentScoreRecordService` 接口 + `TournamentScoreRecordClient` FeignClient + Command/Query/View DTO。
2. `silence-content-service`：实现 `TournamentScoreRecordResource`（implements TournamentScoreRecordService）、`TournamentScoreRecordMyBatisRepository`、`TournamentScoreRecordDao`。

### 主要检索维度

- 按 `id` 精确查询
- 按 `eventGameId + participantId + participantType` 查询（可选 `scoreType` 过滤），分页返回

### 删除策略

逻辑删除（`is_deleted = 1`），通过 `AbstractAutoGeneratedIdLogicalDeletionAuditable` 父类字段保障审计。

## Requirements (RFC 2119)

- **Create**: 系统 MUST 支持创建得分记录，`eventGameId`、`participantId`、`participantType`、`scoreType`、`score` 为必填。
- **Read**: 系统 MUST 支持按 `id` 查询；SHOULD 支持按 `eventGameId + participantId` 分页查询；SHOULD 支持可选 `scoreType` 过滤。
- **Update**: 系统 MUST 支持通过 `id` 更新 `score`、`cycleNumber`、`segmentNumber`、`stageNumber`、`groupId`。
- **Delete**: 系统 MUST 使用逻辑删除。
- **EnumMapping**: `participantType` MUST 以 `tinyint(3) unsigned` 存储；`scoreType` MUST 以 `tinyint(3) unsigned` 存储。
- **Audit**: 系统 MUST 维护 `created_by`、`created_date`、`updated_by`、`updated_date`、`is_deleted` 审计字段。

## Acceptance Criteria (Given/When/Then)

### Scenario 1: 记录场次得分

Given 参赛单位已报名，当前场次已结束。

When 系统调用创建接口，传入 `eventGameId`、`participantId`、`participantType=USER`、`scoreType=STAGE`、`stageNumber`、`score`。

Then 得分记录写入成功，返回新记录 id。

### Scenario 2: 查询参赛单位得分明细

Given 参赛单位在该玩法下有多条得分记录。

When 按 `eventGameId + participantId` 分页查询。

Then 返回分页结果，每条记录包含 `scoreType`、`score`、`stageNumber`/`segmentNumber`/`cycleNumber`。

### Scenario 3: 按得分类型过滤查询

Given 参赛单位有 STAGE、SEGMENT、CYCLE 多种类型得分。

When 按 `eventGameId + participantId + scoreType=CYCLE` 查询。

Then 仅返回 `scoreType=CYCLE` 的得分记录。

### Scenario 4: 更新得分

Given 得分记录存在（id 已知）。

When 调用更新接口传入新的 `score`。

Then 记录的 `score` 字段更新成功，审计字段 `updated_by`/`updated_date` 同步更新。

### Scenario 5: 逻辑删除得分记录

Given 得分记录存在。

When 调用删除接口。

Then 该记录 `is_deleted=1`，查询接口不再返回该条记录。
